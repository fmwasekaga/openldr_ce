# P2-DASH — Custom Dashboards (Metabase-like) Design

**Date:** 2026-06-14
**Sub-project:** P2-DASH (new Phase-2 sub-project)
**Branch:** `feat/p2-dashboards` (off `main`)
**Status:** Approved design — pending implementation plan

## Goal

Replace the static report-card grid on the OpenLDR CE Dashboard page with a
configurable, Metabase-like dashboard system, porting the feature set the team
built in **corlix**: a drag-resize widget grid, custom widget creation through a
visual query builder, a gated raw-SQL escape hatch, dashboard-level filter
variables, multiple dashboards with CRUD, auto-save, and per-widget/per-dashboard
refresh intervals.

## Context & constraints

corlix's dashboard is an **Electron** app: it runs user-typed SQL through
`better-sqlite3` in the main process via IPC, with a CodeMirror SQL editor,
`react-grid-layout`, `recharts`, and a `zustand` store. OpenLDR CE is a different
shape and imposes constraints the port must respect:

- **Browser SPA + Fastify API**, not Electron/IPC. A query engine must live in a
  backend service, not the client.
- **No hand-written SQL** is a core repo convention — every query is generated by
  the Kysely builder.
- The analytics **warehouse can be Postgres *or* SQL Server**; a raw SQL string
  typed by a user will not run unchanged on both dialects.
- The warehouse holds **clinical / PHI data** — arbitrary `SELECT` is a real
  security surface.
- **Auth is a non-functional stub** today (`verifyToken` throws "not
  implemented"); there is a user model but no working login to gate per-user
  dashboards or admin-only features.

## Decisions

| Decision | Choice |
| --- | --- |
| Scope | New sub-project (P2-DASH), own branch off `main`, full corlix feature parity |
| Query layer | **Hybrid** — visual builder (default, Kysely-compiled) + **gated** raw-SQL escape hatch |
| Ownership | **Shared/global** dashboards for now; `owner_id` column reserved (defaulted NULL) for per-user later |
| Raw-SQL gate (interim) | Config flag + Postgres-only + read-only transaction + statement timeout/row cap; admin-role check added when OIDC lands |
| Client state | Add `zustand` (scoped to this feature) |
| Domain placement | New `packages/dashboards` domain package |

## Architecture

New domain package **`packages/dashboards`** (hexagonal: domain logic lives in
`packages/`, never `apps/`). It owns:

- The dashboard / widget / query **types**.
- The **query-model registry** (which `ExternalSchema` tables are queryable, plus
  their dimensions and metrics).
- The **query compiler** (`builder` config → Kysely query against
  `ExternalSchema`), returning the existing `ReportResult` shape so the
  frontend's `ReportView` / recharts render widget data unchanged.
- A **`DashboardStore`** interface (CRUD), implemented over the **internal
  Postgres** DB (always available), wired in `bootstrap` and exposed on
  `AppContext` the way `reporting` is.
- The **gated SQL runner**.

Dependency boundaries (enforced by `.dependency-cruiser.cjs`): `dashboards` may
import `db` / `reporting` types only — never adapters, never `apps`.

### Persistence

One table in the **internal DB** (new migration
`packages/db/src/migrations/internal/011_dashboards.ts`) — documents with
embedded JSON, like corlix:

| column | type | notes |
| --- | --- | --- |
| `id` | text PK | uuid |
| `owner_id` | text NULL | reserved for per-user; NULL = shared/global now |
| `name` | text | |

### Types

```ts
type WidgetType =
  | 'kpi' | 'line-chart' | 'bar-chart' | 'area-chart' | 'row-chart'
  | 'pie-chart' | 'scatter-plot' | 'funnel' | 'progress-bar' | 'gauge'
  | 'table' | 'traffic-light';

interface WidgetConfig {
  id: string;
  type: WidgetType;
  title: string;
  query: WidgetQuery;
  refreshIntervalSec: number;
  visual: WidgetVisual;       // color, axis keys, thresholds, columns, etc. (corlix parity)
}

type WidgetQuery =
  | { mode: 'builder'; model: string; metric: Metric; dimension?: Dimension;
      filters: QueryFilter[]; variableBindings?: Record<string, string> }
  | { mode: 'sql'; sql: string; variableBindings?: Record<string, string> }; // gated
```

The discriminated union is the key adaptation of corlix's loose
`dataSourceId + params`: `builder` compiles to Kysely; `sql` is the gated escape
hatch.

## Query & builder layer

### Model registry (`packages/dashboards/src/models/`)

Each queryable model is a declarative descriptor over an `ExternalSchema` table.
It drives the builder's dropdowns and the compiler's output.

```ts
interface QueryModel {
  id: string;                 // 'service_requests'
  label: string;              // 'Test Orders'
  table: keyof ExternalSchema;
  dimensions: Dimension[];    // groupable / filterable columns
  metrics: Metric[];          // count, count_distinct(col), sum/avg(numericCol)
}
interface Dimension {
  key: string; column: string; kind: 'string' | 'date' | 'number';
  dateGrain?: ('day' | 'week' | 'month' | 'year')[]; // date cols offer truncation
}
interface Metric {
  key: string; label: string;
  agg: 'count' | 'count_distinct' | 'sum' | 'avg' | 'min' | 'max';
  column?: string;
}
```

Seeded from the curated external tables the code-defined reports already query
(`service_requests`, `observations`, `patients`, `diagnostic_reports`,
`specimens`, `organizations`, `locations`) — exposing only known-safe columns,
not the raw schema.

### Builder semantics

A builder widget is `{ model, metric, dimension?, filters[] }`:

- **metric** → the aggregate (the KPI number, or the Y of a chart).
- **dimension** → the group-by (the X / category / table rows). Date dimensions
  add a grain (month/week/…).
- **filters** → `[{ dimension, op, value }]` with a typed op set per kind
  (`eq/in/contains` for strings; `gte/lte/between` for date & number).

### Compiler (`packages/dashboards/src/compile.ts`)

Pure function `(QueryModel, WidgetQuery) → Kysely query`. It references only
columns named in the registry — anything else is rejected before a query is
built. Returns the existing `ReportResult` (`columns / rows / chart`) so widget
rendering reuses `ReportView`. Grouping/filtering happen in Kysely; any
bucketing/pivoting happens in pure JS helpers (repo convention). Portable across
PG/MSSQL because it is all builder-generated.

### Dashboard filters & variables

Dashboard-level `filters[]` (text / number / date / date-range, with static or
SQL-sourced options) bind into widgets via `variableBindings`. In **builder**
mode a binding overrides a filter's value; in **sql** mode it substitutes a
`{{var}}` placeholder. Bound values are always passed as **parameters**, never
string-concatenated.

### Gated SQL escape hatch (`packages/dashboards/src/sql-runner.ts`)

1. Enabled only when `DASHBOARD_SQL_ENABLED=true` **and**
   `TARGET_STORE_ADAPTER=pg` (new config flags in `packages/config`).
2. Statement parsed/validated: single statement; must be `SELECT` / `WITH`;
   reject DML/DDL, multiple statements, comment smuggling.
3. Executed in a **read-only transaction** (`SET TRANSACTION READ ONLY`) with a
   **statement timeout** and a **row cap**.
4. `{{var}}` placeholders replaced with bound **parameters** (not interpolated).
5. When auth lands, add the admin-role check as a 4th gate (structured as a
   one-line addition).

If SQL mode is disabled, the UI hides the editor tab and the route rejects
`mode:'sql'` payloads server-side (defense in depth).

## API

New `apps/server/src/dashboards-routes.ts`, wired through `bootstrap` like
reports:

- `GET    /api/dashboards` — list
- `GET    /api/dashboards/:id` — get
- `POST   /api/dashboards` — create
- `PUT    /api/dashboards/:id` — save (layout/widgets/filters/name/refresh)
- `DELETE /api/dashboards/:id` — delete
- `POST   /api/dashboards/query` — run a widget query (builder or gated sql),
  returns `ReportResult`
- `GET    /api/dashboards/models` — model registry for the builder UI

Reuse the reports routes' error mapping (404/400/500) so DB errors never leak raw
SQL/schema to the client.

## Frontend (`apps/web`)

New deps (scoped to this feature): `react-grid-layout` (+ CSS), `zustand`,
CodeMirror 6 (`codemirror`, `@codemirror/lang-sql`, `@codemirror/view`,
`@codemirror/state`, theme). `recharts` (2.13), shadcn, and lucide are already
present and reused.

Structure under `apps/web/src/dashboard/`:

- `store.ts` — zustand store: current dashboard, `widgets`, `layout`, `filters`,
  `widgetData` cache, `editing` flag, refresh trigger (mirrors corlix
  `dashboard-store`).
- `DashboardPage.tsx` — replaces today's report-card grid. Dashboard selector +
  CRUD (create/rename/duplicate/delete/import/export JSON), edit-mode toggle,
  refresh-interval control, debounced auto-save while editing.
- `DashboardGrid.tsx` — `react-grid-layout`, 12-col / 80px rows / vertical
  compaction; drag+resize only in edit mode; `.drag-handle` on widget headers;
  persists `{ x, y, w, h }` to the store.
- `DashboardWidget.tsx` — fetches via `/api/dashboards/query`, handles per-widget
  refresh + variable binding, routes config → renderer.
- `widgets/` — the **12 renderers** (full parity): `Kpi`, `LineChart`,
  `BarChart`, `AreaChart`, `RowChart`, `PieChart`, `ScatterPlot`, `Funnel`,
  `ProgressBar`, `Gauge`, `Table`, `TrafficLight`. Charts reuse recharts;
  KPI/Gauge/Progress/TrafficLight are small SVG/CSS like corlix. Default sizes
  match corlix (KPI 3×2, charts 6×4, pie/funnel 4×4, gauge 3×3).
- `editor/WidgetEditorDialog.tsx` — full-screen shadcn `Dialog`: pick type →
  **Builder tab** (model / metric / dimension+grain / filters, dropdowns driven
  by `/api/dashboards/models`) **or SQL tab** (CodeMirror, shown only when SQL
  mode is enabled) → **live preview** (debounced `/api/dashboards/query`) →
  visual config (color, axes, thresholds, columns…) → save.
- `DashboardFilterBar.tsx` / `DashboardFilterEditor.tsx` — dashboard-level filter
  controls + their definition dialog.

API client additions in `apps/web/src/api.ts`:
`listDashboards/getDashboard/saveDashboard/createDashboard/deleteDashboard`,
`runWidgetQuery(query, filterValues)`, `listModels()`.

Routing: `/` (or `/dashboard`) → `DashboardPage`; existing report links keep
working (reports stay a separate page). Today's `Dashboard.tsx` report grid is
superseded — its "list of reports" moves into the Reports area so nothing is
lost.

Seed: one default dashboard (`is_default`) provisioned on first load with a few
builder widgets over real models (e.g. test-volume KPI + orders-trend line +
category bar) so the page isn't empty — analogous to corlix's `DEFAULT_WIDGETS`.

## Security

- **Builder mode** is the safe default: only registry-named columns/aggregates
  reach Kysely; all values parameterized. No user string ever becomes SQL.
- **SQL mode** defense-in-depth: config-flag + Postgres-only gate → SELECT/WITH
  parse validation → read-only transaction + statement timeout + row cap →
  parameterized variables. UI hides the tab when disabled; the route also rejects
  `mode:'sql'` server-side. Role-gate slot reserved for OIDC.
- **No PHI in widget configs** — dashboards store queries, not result rows;
  result rows are fetched live per viewer.
- Reuse reports' error mapping so DB errors never leak raw SQL/schema.

## Testing

- `packages/dashboards`: compiler unit tests (each metric/dimension/grain/filter
  → expected SQL via Kysely `.compile()`), registry validation, SQL-validator
  tests (reject DML / multi-statement / comment smuggling; accept SELECT/WITH),
  store CRUD tests.
- `apps/server`: route tests for `/api/dashboards*`, including the SQL-disabled
  rejection path.
- `apps/web`: store reducer tests, a render test per widget type, editor
  builder→preview flow test.
- **e2e** (`e2e/tests/dashboard.spec.ts`): create dashboard → add a builder
  widget → see it render → drag/resize persists across reload. Committed
  screenshot pass like the docs work.
- Gates: `turbo typecheck lint test build` and **dependency-cruiser** must pass.

## Rollout / sequencing

1. `packages/dashboards` types + model registry + builder compiler (+ tests).
2. Internal-DB migration + `DashboardStore` + bootstrap wiring.
3. `/api/dashboards*` routes (+ tests).
4. Frontend: store, grid, widget renderers, `DashboardPage`, API client.
5. Widget editor — builder tab + live preview.
6. Dashboard-level filters + variable bindings.
7. Gated SQL escape hatch (config flags, sql-runner, editor SQL tab).
8. Default-dashboard seed, e2e + screenshots, docs page entry.

## Conventions

- Branch `feat/p2-dashboards` off `main`.
- Commit format `<type>(<scope>): <desc> (P2-DASH)`; **no Co-Authored-By
  trailer**; gpg-sign off.
- ESM, `import type`, no file extensions on local imports, barrel `src/index.ts`,
  strict TS — matching the rest of the monorepo.

## Out of scope (deferred)

- Real OIDC auth / true per-user ownership (owner_id reserved, not activated).
- Cross-database raw SQL (SQL mode is Postgres-only by design).
- Scheduled/emailed dashboard exports.
- Dashboard sharing/permissions beyond the global set.
